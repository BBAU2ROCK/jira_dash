## 2. 화면 구성

### Section 1 — 백로그 상태 카드 (6개)
| 카드 | 의미 |
|------|------|
| 잔여 (전체) | leaf 이슈 합계 (filterLeafIssues 룰 적용) |
| 활성 | done/취소 제외 — 진행해야 할 작업 |
| 보류 | status='보류' — 별도 카운트 |
| 미할당 | assignee 없음 — 그루밍 필요 |
| 90일 완료 | 최근 90일 내 해결된 이슈 |
| 마감일 없음 | 활성 중 duedate 누락 |

### Section 2 — 오늘/이번주 + 지연
- **오늘 완료**: 한국 시간(KST) 기준
- **이번주 완료**: 월요일~일요일
- **미완료 지연**: 지금 처리 필요한 이슈 ⚠
- **완료 지연**: 이미 완료됐지만 마감일 초과 (회복 완료, 참고)

### Section 3 — 일별 완료 추이
최근 30일 BarChart. 일평균·최대값·합계 표시.

### Section 4 — 팀 ETA (3 시나리오)
| 시나리오 | 가정 | 사용 |
|---------|------|------|
| 낙관 | 모든 일이 누구에게나 재할당 가능 | 비현실적 — 참고만 |
| **기준 ★** | 현재 할당 유지 (max 개인 ETA) | **권장 약속일** |
| 병목 | 가장 ETA 큰 인원 | 위험 신호 |

### Section 5 — 확률 분포 (Funnel)
| 백분위 | 의미 | 사용 |
|--------|------|------|
| P50 | 50% 확률 (중앙값) | 동전 던지기 — 약속하지 말 것 |
| **P85** | 85% 확률 | **권장 약속일** |
| P95 | 95% 확률 | 위험 최소화 약속 |

### Section 6 — 공수 분석
- **총 공수**: 인시 + 인일 환산 + 신뢰 구간
- **데이터 출처 분포**: Worklog / SP / 난이도 / Cycle time fallback
- **팀 capacity**: 인원 × 8h × 가동률 65%

### Section 7 — ETA-공수 상호 검증
두 모델 격차 30% 이상이면 경고:
- "처리량 ETA가 공수보다 김" → 블로커·대기 시간 큼 의심
- "공수가 처리량 ETA보다 큼" → Worklog 미기록 가능성

### Section 8 — 담당자별 표
정렬 가능 (가나다 default). 활동 7일 미만 인원은 회색 처리 (휴가 가능성).

## 4. 신뢰도 등급 — 정직성 원칙

| 등급 | 표시 |
|------|------|
| ●●● **높음** | 단일 날짜 + 범위 + 분포 모두 |
| ●●○ **중간** | P85 단일 날짜 + 신뢰 구간 |
| ●○○ **낮음** | **범위만**, 단일 날짜 숨김 |
| ○○○ **예측 불가** | 진단 정보만, 모든 ETA 숨김 |

> "정확한 단일 날짜는 거짓말. 범위가 진실. 확률 분포가 더 정직한 진실."

낮은 신뢰도일 때 단일 날짜를 보여주는 것이 가장 큰 거짓말이라는 원칙으로 의도적 정보 숨김.

## 5. FAQ

### Q1. ETA가 매주 후퇴하는 이유?
**A.** Scope creep — 신규 이슈가 완료보다 빠르게 추가되고 있습니다. Section 4 위의 "Scope 상태" 알림 확인. 신규 차단 또는 인력 보강 필요.

### Q2. 왜 단일 날짜를 안 보여주나?
**A.** 통계적 신뢰도가 낮아 단일 날짜를 약속하면 거짓말이 됩니다. 범위 또는 P85 약속일을 사용하세요.

### Q3. 동료 ETA가 매우 길게 나오는데?
**A.** 다음을 확인:
- 활동 일수 (7일 미만이면 회색 처리, 통계 부정확)
- 휴가/병가 기간 — 시스템에 정보 없음
- 도메인 specialization (백/프) — 모델 미반영

성과 평가 용도가 아닌 워크로드 균형 분석 용도입니다.

### Q4. 공수와 ETA가 다른데?
**A.** 두 모델이 각각 다른 가정을 사용합니다:
- ETA = 처리량 (블로커·대기 포함된 wall-clock)
- 공수 = 작업의 양 (focused work time)
30% 이상 격차 시 경고 표시. "프로세스 비효율" 또는 "공수 미기록" 해석 함께 표시.

### Q5. SP/난이도가 안 보이는데?
**A.** Phase 0 측정 결과 본 환경에서 SP·난이도 커버리지가 0%입니다. Worklog 또는 Cycle time fallback만 활성됩니다 — 환경에 맞게 자동 비활성.

### Q6. 미할당 이슈 처리는?
**A.** 별도 카운트만 하고 ETA에는 포함하지 않습니다. 그루밍 후 할당하세요.

### Q7. 백로그가 거의 비어있는데 (예: IGMU)?
**A.** "🎉 활성 백로그가 비어 있습니다" 안내가 표시됩니다. 다른 활성 프로젝트(예: IPCON)를 선택하세요.

### Q8. 한국 공휴일 반영?
**A.** 영업일 계산에 `JIRA_CONFIG.KOREAN_HOLIDAYS_2026` 적용. 매년 이 배열 갱신 필요.

### Q9. 캐싱 정책?
**A.** 프로젝트 이슈는 5분 staleTime. 우상단 새로고침 버튼으로 강제 fetch 가능.

### Q10. 방법론 자세히?
**A.** 화면 우상단 **[방법론 보기]** 버튼 → 모달로 전체 산식 설명. 또는 `docs/progress-prediction-analysis.md` 참조.

## 6. 트러블슈팅

### 데이터 로드 실패 (401)
- 우상단 [설정] → Jira 자격증명 확인
- 사용자가 해당 프로젝트 접근 권한 있는지 확인

### "예측 불가" 가 계속 표시
다음 중 하나:
- 활동 일수 < 7일 (신규 프로젝트)
- Scope 발산 (신규/완료 1.5x 초과)
- 백로그 0건

### 차트가 비어있음
참조 기간(default 30일) 내 완료 이슈가 없음. 더 긴 history는 Tier 3에서 제공 예정.

## 7. 한계와 가정 (정직 안내)

본 예측은 다음 가정을 사용하며, 실제와 다를 수 있습니다:
- 주말 + 한국 공휴일 제외 (영업일 기준)
- 휴가/병가 정보 없음
- 도메인 specialization 미반영
- 팀 인원 변동 미반영 (신규 합류 인원은 30일 미만이면 confidence=low)
- Scope creep 보정 ON (ETA 후퇴 가능)
- 가동률 65% 가정 (변경 옵션은 Tier 3)
