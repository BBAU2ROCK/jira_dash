# 프로젝트 현황: 할 일 전용 업무로그 담당자별 반영 분석

## 1. 요청 사항 요약

- **상황**: 하위 작업(Subtask)에는 업무로그가 없고, **할 일(Parent 이슈)**에만 업무로그가 기록된 경우가 있음.
- **요청**: 해당 업무로그를 **담당자별 업무로그**에 추가해서 카운트할 수 있는지 여부 분석, 및 소스 수정 시 문제 여부 분석.

---

## 2. 현재 동작 요약

### 2.1 건수 규칙 (리프 이슈)

- **규칙** (`filterLeafIssues`, `jira-helpers.ts`):
  - 할 일만 있는 경우: 해당 이슈를 통계에 포함.
  - 하위 작업이 있는 경우: **(부모) 할 일은 제외**, 하위 작업만 통계에 반영.
- **결과**: 프로젝트 현황의 “전체/완료/진행/대기/지연/조기완료” 및 **담당자별 건수·업무로그**는 모두 **리프 이슈(leaf issues)** 기준으로만 계산됨.

### 2.2 담당자별 업무로그 집계 (현재)

- **위치**: `src/components/project-stats-dialog.tsx`
- **로직**:
  - `leafIssues`만 순회.
  - 각 이슈의 `issue.fields.timespent`(초 단위 합계)로:
    - `timeSpent > 0` → 해당 담당자의 `withWorklog` 건수 +1, `totalTimeSpent`에 가산.
    - `timeSpent === 0` → `withoutWorklog`에 포함.
- **한계**: 하위 작업이 있는 “할 일”은 리프에서 제외되므로, **할 일에만 기록된 업무로그는 담당자별 통계에 전혀 반영되지 않음.**

## 3. 가능 여부 분석

### 3.1 데이터·API 측면

| 항목 | 내용 |
|------|------|
| **데이터 존재 여부** | `getIssuesForEpic` 등으로 **부모(할 일)와 하위 작업 모두** 조회하며, 요청 필드에 `timespent`, `worklog` 포함됨. |
| **부모 이슈 식별** | `issue.fields.subtasks?.length > 0` 또는 `jira-helpers`와 동일하게 “자식이 있는 부모 키 집합”으로 식별 가능. |
| **업무 시간 값** | 이슈 단위 집계 값 `fields.timespent`(초)가 이미 있으므로, 별도 worklog 목록 합산 없이 사용 가능. |
| **담당자** | 부모 이슈의 `fields.assignee?.displayName` (없으면 `미할당`). |

**결론**: 할 일(부모)에만 있는 업무로그를 “담당자별”로 추가 집계하는 데 필요한 데이터는 **이미 모두 확보되어 있음**. API 추가 호출 없이 구현 가능.

### 3.2 로직·규칙 측면

- **건수 규칙 유지**: 전체/완료/진행/대기/지연 등 **이슈 건수**는 기존대로 **리프만** 사용하면 됨. “할 일에만 로그 있음”은 **업무 시간 합계**에만 반영하고, 건수 분류는 기존과 동일하게 둘 수 있음.
- **담당자별 집계**:
  - **옵션 A (권장)**: “할 일에만 있는 업무로그”의 **시간만** 담당자별 `totalTimeSpent`에 가산.  
    - “로그 있음/없음” **건수**는 리프 이슈 기준 유지 (테이블 의미 일관성 유지).
  - **옵션 B**: 위 시간 가산에 더해, 해당 “할 일” 1건을 담당자별 “로그 있음” 건수에 포함.  
    - 이 경우 “로그 있음” = “리프 중 로그 있는 건수 + (해당 담당자의) 할 일만 로그 있는 건수”가 됨.

## 3.3 “로그 없음” 건수에 포함하는 것이 맞는지 분석

**질문**: 할 일에만 업무로그가 있고 하위 작업에는 없는 경우, 그 **건수**를 담당자별 **“로그 없음”**에 포함하는 것이 맞는가?

**결론: 맞지 않음. “로그 없음”에 포함하면 안 됨.**

### 이유 1 — 의미상 부적절

- **“로그 없음”**은 “업무로그가 **없는** 이슈”의 건수를 의미함.
- 해당 케이스에서 **할 일(부모) 이슈에는 업무로그가 있음** (timespent > 0).
- 따라서 그 **할 일**을 “로그 없음”으로 분류하면, “업무로그가 있는 이슈를 로그 없음으로 세는 것”이 되어 **의미상 잘못**임.
- “하위 작업에 로그가 없다”는 사실은, 이미 **하위 작업(리프)** 단위로 각 담당자 행의 “로그 없음”에 반영됨 (하위 이슈가 리프이고 timespent=0이면 해당 담당자의 `withoutWorklog`에 포함됨).

### 이유 2 — 건수 일관성

- 담당자 행의 **“전체”**는 **리프 이슈만** 사용함 (건수 규칙).
- 현재 **로그 있음 + 로그 없음 = 전체** 가 성립함 (같은 리프 집합을 로그 유무로 이분).
- 할 일(부모)을 “로그 없음”에만 넣으면:
  - 해당 담당자의 **전체**에는 포함하지 않으므로, **로그 있음 + 로그 없음 ≠ 전체** 가 되어 **일관성이 깨짐**.
- 반대로 부모를 “전체”에도 넣으면, “전체/완료/진행/대기” 등 **모든 건수 규칙**을 리프만 쓰는 현재 원칙과 충돌함.

### 이유 3 — 올바른 반영 방식 (현재 적용)

- 할 일에만 업무로그가 있는 경우:
  - **기록 시간(totalTimeSpent)** 에만 반영하는 것이 적절함 (이미 적용됨).
  - **건수**는 리프만 유지: “로그 있음/없음” 모두 **리프 이슈**만 세어, 의미와 일관성을 유지함.

**정리**: “할 일에만 로그 있고 하위에는 없는 경우”는 **로그 없음 건수에 포함하지 않는 것**이 맞고, 현재처럼 **기록 시간에만 반영**하는 방식이 타당함.

## 3.4 할 일 업무로그를 근거로 하위 작업의 “로그 없음”을 “로그 있음”으로 처리하는 것이 맞는지 분석

**질문**: 하위 작업이 있는 할 일에만 업무로그가 있고, 하위 작업에는 업무로그가 없는 경우, **할 일에만 있는 업무로그의 존재**를 근거로 하위 작업의 “로그 없음”을 “로그 있음”으로 처리(집계)하는 것이 맞는가?

**결론: 일반적으로 맞지 않음. “하위 작업 = 로그 있음”으로 바꾸어 집계하는 것은 권장하지 않음.**

### 전제 정리

- **할 일(부모)**: 업무로그 있음 (timespent > 0).
- **하위 작업(리프)**: 각각 업무로그 없음 (timespent = 0) → 현재는 담당자별 “로그 없음” 건수에 포함됨.
- **제안**: “부모에 로그가 있으니, 그 부모의 하위 작업들은 통계상 ‘로그 있음’으로 간주한다.”

### 이유 1 — 데이터·시스템과의 불일치

- Jira에서 **업무로그는 이슈 단위**로 기록됨. 각 이슈(할 일/하위 작업)마다 별도 worklog.
- 하위 작업에는 **실제로 업무로그가 없음** (timespent = 0, worklog 없음). 시스템 상태와 동일한 의미를 쓰는 것이 맞다면, “로그 있음”은 “**그 이슈에** 업무로그가 있는 경우”에만 써야 함.
- 할 일의 업무로그 “내용”을 근거로 하위 작업을 “로그 있음”으로 바꾸면, **Jira에 없는 상태를 있는 것처럼 표시**하는 것이 되어, 데이터 정합성·신뢰도가 떨어짐.

### 이유 2 — 지표 의미의 변경

- 현재 **“로그 있음”** = “해당 **리프 이슈**에 업무로그가 기록된 건수”.
- 제안을 적용하면 **“로그 있음”** = “해당 리프 이슈에 업무로그가 있음 **또는** 그 리프의 상위 할 일에 업무로그가 있음”으로 **정의가 바뀜**.
- 같은 용어로 두 가지 정의를 쓰면 해석이 혼란스럽고, “로그 있음” 클릭 시 보이는 이슈 목록(리프) 중에는 **실제로 로그가 없는 이슈**가 섞이게 됨.

### 이유 3 — 담당자별로 보았을 때 혼란

- **할 일 담당자 A, 하위 작업 담당자 B/C** 인 경우:
  - 할 일의 업무로그·기록 시간은 **A**에게만 totalTimeSpent로 반영하는 것이 현재 방식이고 타당함.
  - 여기서 “하위 작업을 로그 있음으로 처리”하면, **B·C**의 “로그 있음” 건수에 각각 그 하위 작업이 포함됨.
  - 하지만 그 하위 이슈에는 B·C 본인이 기록한 업무로그가 없고, **기록 시간도 0**임. “로그 있음”인데 해당 이슈의 기록 시간이 0으로 나오면 의미가 애매해짐.
- 즉, **“로그 있음” 건수만 올리고, 그 이슈의 시간·내용은 부모 것**이라는 구조가 되어, 담당자별 통계 해석이 어려워짐.

### 이유 4 — “내용을 토대로”의 한계

- “할 일에만 있는 업무로그의 **내용**을 토대로” 하위를 로그 있음으로 한다는 것은, 내용이 어떤 하위에 대응하는지 **자동으로 매핑하기 어렵다**는 문제가 있음. 업무로그 코멘트만으로 “이 시간은 하위 작업 1번, 이 시간은 2번”이라고 나누는 것은 보통 불가능에 가깝고, 구현해도 휴리스틱에 의존하게 됨.
- 따라서 현실적으로는 “할 일에 로그가 있으면 **그 할 일의 모든 하위**를 로그 있음으로 간주”하는 식이 될 가능성이 큼. 그렇게 하면 위의 “데이터 불일치·지표 의미 변경·담당자별 혼란”이 그대로 발생함.

### 대안으로 타당한 것 (현재 방식 유지)

- **할 일에만 업무로그가 있는 경우**
  - **기록 시간**: 할 일 담당자에게만 totalTimeSpent로 반영 (현재 구현). ✅
  - **로그 있음/없음 건수**: 각 **리프 이슈**의 실제 업무로그 유무만 사용.  
    → 하위 작업은 그대로 “로그 없음”으로 두는 것이, Jira 데이터와 일치하고 지표 의미도 명확함.

### 정리

- 할 일에만 있는 업무로그를 **근거**로 하위 작업의 “로그 없음”을 “로그 있음”으로 **바꾸어 집계하는 것은**, 데이터 정합성·지표 정의·담당자별 해석 측면에서 **맞지 않음**.
- **권장**: 하위 작업은 **해당 이슈에 실제로 업무로그가 있는지**만 보고 “로그 있음/없음”을 구분하는 현재 방식을 유지하는 것이 맞음.

## 4. 구현 방안 (소스 수정)

### 4.1 수정 위치

- **파일**: `src/components/project-stats-dialog.tsx`
- **대상**: 담당자별 통계를 채우는 루프 및 그 직전에 “부모(할 일) 전용 업무로그” 반영.

### 4.2 구체적 로직

1. **리프 제외 부모 키 집합 계산**  
   - `issues` 기준으로 `filterLeafIssues`와 동일한 규칙으로 “하위가 있는 부모” 키 집합 생성  
   - 예: `parentsWithChildren = new Set( issues.filter(i => i.fields.subtasks?.length > 0).map(i => i.key) )`

2. **기존 담당자별 집계**  
   - 현재처럼 `leafIssues`만 순회하여 `assigneeMap`에 total/done/inProgress/todo/delayed/earlyDone/compliant, **그리고** withWorklog/withoutWorklog/totalTimeSpent 채우기.

3. **할 일 전용 업무로그 추가 반영**  
   - `issues`를 순회:
     - `parentsWithChildren.has(issue.key)` 이고  
     - `(issue.fields.timespent || 0) > 0` 인 이슈만 대상.
   - 해당 이슈의 담당자 `displayName`(없으면 `'미할당'`)에 대해:
     - **공통**: `assigneeMap.get(name).totalTimeSpent += issue.fields.timespent`
     - **옵션 B 적용 시**: `withWorklog`에 해당 부모 이슈를 넣어 “로그 있음” 건수에 포함 (별도 배열로 관리하거나, 기존 withWorklog에 부모 이슈를 넣을지 팀 정책에 따라 결정).

### 4.3 UI 변경

- **옵션 A**: “기록 시간” 열만 변경됨. 할 일 전용 업무로그가 있으면 해당 담당자의 **기록 시간**이 기존보다 늘어남. “로그 있음/없음” 건수·퍼센트는 리프만 기준으로 유지.
- **옵션 B**: “로그 있음” 건수도 해당 담당자에서 증가. 툴팁이나 라벨로 “리프 + 부모만 로그” 구분 표시를 넣을 수 있음.

## 5. 소스 수정 시 문제 여부

### 5.1 부작용·일관성

| 항목 | 판단 |
|------|------|
| **건수 규칙** | 리프만 쓰는 규칙은 그대로 두고, “시간”만 보조적으로 부모에서 가져오므로 기존 KPI/건수와 충돌 없음. |
| **중복 집계** | 부모는 리프에 포함되지 않으므로, 부모의 timespent를 한 번만 해당 담당자에게 더하면 됨. 중복 없음. |
| **데이터 소스** | `issues`는 이미 대화상자에 전달되는 props. 부모/하위 모두 포함되어 있어 추가 API 불필요. |
| **assignee 없음** | 부모의 assignee가 없으면 `미할당`으로 집계. 기존 리프 “미할당”과 동일한 버킷 사용. |

### 5.2 엣지 케이스

- **부모만 있고 하위가 0개**: 해당 이슈는 리프로 포함되므로, 기존 로직에서 이미 담당자별 업무로그에 포함됨. 부모 전용 로직에서는 “하위가 있는 부모”만 대상으로 하므로 **이중 집계되지 않음**.
- **같은 담당자가 리프 + 부모 전용 로그 보유**: totalTimeSpent에만 더해지므로 수치상 문제 없음.

### 5.3 정리

- **가능 여부**: **가능**. 이미 보유한 데이터와 기존 규칙만으로 구현 가능.
- **소스 수정 시 문제**:  
  - **옵션 A(시간만 반영)**로 적용 시 기존 건수·KPI와의 일관성을 해치지 않으며, 수정 범위도 작음.  
  - **옵션 B**는 “로그 있음” 건수 정의를 확장하는 것이므로, 요구사항과 팀 합의에 따라 적용 여부를 결정하면 됨.

## 6. 권장 사항

1. **1단계**: 옵션 A만 적용하여 “할 일에만 있는 업무로그”를 **담당자별 기록 시간(totalTimeSpent)**에만 반영.  
   - “로그 있음/없음” 건수는 리프 기준 유지.  
   - 사용자 요청의 “담당자별 업무로그로 추가해서 카운트”를 “시간 기준 카운트”로 충족.
2. **2단계(선택)**: 필요 시 “할 일에만 로그 있음” 건수를 별도 열이나 툴팁으로 노출(옵션 B 확장).

이 순서로 적용하면 소스 수정 시 기존 동작을 깨지 않으면서 요청 기능을 반영할 수 있음.

## 7. 사이드이펙트 검증 (재검토)

### 7.1 부모 집합 일치

- **검증**: "리프에서 제외되는 부모" 집합을 `filterLeafIssues`(jira-helpers)와 **동일 규칙**으로 계산함.
- **규칙**: (1) `issue.fields.parent?.key`가 있으면 그 부모 키 추가, (2) `issue.fields.subtasks?.length > 0`이면 해당 이슈 키 추가. → 동일 Set 사용으로 이중 집계·누락 방지.

### 7.2 건수·KPI·정렬

| 항목 | 검증 결과 |
|------|-----------|
| **건수** | 전체/완료/진행/대기/지연/조기완료는 `leafIssues`만 사용. 수정 로직은 `totalTimeSpent`만 변경. **영향 없음.** |
| **KPI** | `assigneesWithKPI`는 `assignees`의 `a.total`(리프만)으로 `calculateKPI(a.total)` 호출. 부모 전용 담당자는 `total: []` → `calculateKPI([])`는 0/0/D등급 반환. **크래시 없음.** |
| **정렬** | `assignees` 정렬은 `b.total.length - a.total.length`. 부모 전용 담당자는 total.length=0 → 목록 하단에 표시. **의도된 동작.** |

### 7.3 UI·엣지 케이스

| 항목 | 검증 결과 |
|------|-----------|
| **로그 있음/없음** | 리프 기준 유지. 부모 전용 담당자는 0건 / 0건, 기록 시간만 양수. **일관성 유지.** |
| **openGroup** | "로그 있음" 클릭 시 `a.withWorklog` 사용. 부모 전용 담당자는 빈 배열 → 버튼 비활성. **문제 없음.** |
| **할 일만 있고 하위 0개** | 해당 이슈는 리프에 포함되며 `parentsWithChildren`에 들어가지 않음. **이중 집계 없음.** |
| **assignee 없음** | 부모의 assignee 없으면 `미할당`으로 집계. 기존과 동일. |

### 7.4 결론

- 수정 범위는 **담당자별 totalTimeSpent 가산**만이며, 건수·KPI·리프 규칙은 변경하지 않음.
- 부모 집합을 jira-helpers와 동일 규칙으로 계산하도록 반영하여 **사이드이펙트 없음**으로 검증 완료.

## 8. 적용 이력

- **옵션 A 적용**: `project-stats-dialog.tsx`에서 "할 일(부모)에만 업무로그가 있는 경우" 해당 부모의 `timespent`를 해당 담당자별 `totalTimeSpent`에 가산하도록 구현함. "로그 있음/없음" 건수는 리프 기준 유지.
- **사이드이펙트 재검토**: 부모 집합을 `filterLeafIssues`와 동일 규칙(parent.key + subtasks.length>0)으로 계산하도록 수정. 문서에 검증 섹션 추가.
