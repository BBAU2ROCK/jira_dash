## 2. 분석 가능한 데이터 차원

### A. 에픽 단위 메트릭 (회고)

| 총 lead time (epic created → 마지막 task done) | 계산 가능 | 신규 |
| 활성 기간 (첫 in-progress → 마지막 done) | changelog 필요 | 부분 (Tier 3 cycle time) |

| 재작업률 (done → reopen) | changelog status 분석 | 신규 |
| **결함 발생** (epic 내 task → TQ 결함 link) | `epicMappingStore` 활용 | ✅ |

| 메트릭 | 가치 | 위험 |

| 타입별 강점 (story vs task vs bug) | 강점 매핑 | 낮음 |

| 결함 fix lead time | 회복 속도 | 낮음 |

| 협업 패턴 (멘션·댓글) | 소셜 그래프 | **매우 높음** (이번에 X) |

## 3. 분석 레벨 — 5단계 정의

### Level 1 — 단일 에픽 회고 (post-mortem)
- **입력**: 사이드바에서 에픽 1개
- **출력**: KPI + 계획 vs 실제 + lead time + 담당자 기여 + 결함 분포
- **가치 ⭐⭐⭐ · 위험 ⭐ · 공수 1.5d**

### Level 2 — 다중 에픽 비교 (벤치마킹)
- **입력**: 에픽 2~10개
- **출력**: 비교 표 (KPI, cycle time, 결함율, 정시 완료율) + 평균 대비 차이
- **가치 ⭐⭐⭐⭐ · 위험 ⭐⭐ · 공수 1.5d**

### Level 3 — 시계열 추세 (분기 회고)
- **입력**: 90일~12개월 모든 완료 에픽
- **출력**: 분기별 KPI/결함율/cycle time 추세 차트
- **가치 ⭐⭐⭐ · 위험 ⭐⭐ · 공수 2d**
- 전체 프로젝트 fetch 필요 (Tier 2 useProjectIssues 부활)

### Level 4 — 개발자 강점 매트릭스
- **입력**: 모든 완료 task의 (담당자, type, cycle time, 결함)
- **출력**: type × 인원 heatmap, 강점·약점 카드
- **가치 ⭐⭐⭐⭐ · 위험 ⭐⭐⭐⭐⭐ · 공수 3d + 정책 검토**
- **익명화 default ON 권장**

### Level 5 — Anomaly Detection (자동 위험 패턴)
- **입력**: 모든 완료 에픽
- **출력**: ML clustering·classification → "위험 패턴 N개"
- **가치 ⭐⭐⭐⭐⭐ · 위험 ⭐⭐⭐ · 공수 7d+**
- 데이터 양 부족 시 ROI 불확실

## 6. KPI 탭과의 관계

기존 [KPI 성과] 탭은 **점수 산식**(완료율·준수율·조기·결함밀도)에 집중. Tier 4는 **회고/학습 관점**.

권장: **신규 탭 "에픽 회고"** 추가 (4번째)
```
[프로젝트 현황] [KPI 성과] [진행 추이/예측] [에픽 회고 ★NEW]
```

기능 분리 명확. KPI 산식은 변경 X (회고에 인용만).

## 7. 윤리·문화 위험 — Level 4 핵심

| 위험 | 완화 |
|------|------|
| 인사 평가 오용 | "코칭 도구" 명시, 익명화 default ON |
| 결함 책임 추궁 | "결함 = 시스템 신호 (개인 X)" 안내 |

| 신규 인원 ramp-up 미반영 | 합류 N개월 미만 별도 표시 |
| 외부 공유 시 노출 | 익명화 + 스크린샷 가이드 |
| Specialization 미반영 | "도메인 분담 미반영" 노트 |

→ Level 4 진행 전 사내 HR/법무 검토 권장.
