# 프로젝트 통계 – 보류 건을 완료로 표시하는 부분 분석

## 요약

- **보류 건을 완료로 “특별 처리”하는 전용 로직은 없습니다.**
- 완료 여부는 **Jira API의 `status.statusCategory.key === 'done'` 한 가지 기준**으로만 판단합니다.
- 따라서 **Jira 워크플로우에서 “보류” 상태가 Done 카테고리에 속하면** 자동으로 완료로 집계되고, 그렇지 않으면 완료로 집계되지 않습니다.

---

## 1. 현재 “완료” 판단 기준

### 1.1 프로젝트 통계 다이얼로그 (`src/components/project-stats-dialog.tsx`)

| 구분 | 기준 | 코드 위치 |
|------|------|-----------|
| **완료(done)** | `status.statusCategory.key === 'done'` | 59–60행 |
| **진행(inProg)** | `statusCategory.key === 'indeterminate'` | 61행 |
| **대기(todo)** | `statusCategory.key === 'new'` | 62행 |
| **그 외** | 모두 `todo`로 분류 | 110–114행 (else → s.todo) |

- 완료율·파이/막대·담당자별 “완료” 건수·KPI 완료율 등은 모두 위 `done` 집합을 사용합니다.
- **상태 이름(예: "보류", "Done", "Resolved")을 직접 보지 않고**, **카테고리 키(`done` / `new` / `indeterminate`)만** 사용합니다.

### 1.2 KPI 서비스 (`src/services/kpiService.ts`)

- 완료 여부: `issue.fields.status.statusCategory.key === 'done'` (56–57행).
- agreed-delay·verification-delay 등 **라벨**은 “완료로 쳐줄지”가 아니라, **분모 제외·준수 인정** 등에만 사용됩니다.
- **“보류”를 완료로 넣거나 빼는 별도 분기 없음.**

### 1.3 이슈 리스트·필터·기타

- `issue-list.tsx`, `issue-filter-bar.tsx`, `issue-drawer.tsx` 등에서도 “완료”는 모두 `statusCategory.key === 'done'` 기준입니다.
- `jiraConfig.ts`에는 보류 관련 라벨/상태명 설정이 없고, `AGREED_DELAY`·`VERIFICATION_DELAY`만 정의되어 있습니다.

---

## 2. “보류”가 통계에서 어떻게 보이는지

- **Jira에서 “보류” 상태가 Done 카테고리에 매핑된 경우**  
  → API가 `statusCategory.key === 'done'`을 주므로, **별도 코드 없이도 “완료” 건수·완료율·KPI에 포함**됩니다.
- **“보류”가 To Do(`new`) 또는 In Progress(`indeterminate`) 카테고리인 경우**  
  → 완료로 집계되지 않고, 대기 또는 진행으로만 집계됩니다.

즉, “보류 건을 완료로 처리하여 표시”하는 **앱 내 전용 로직은 없고**, 전적으로 **Jira의 status category 설정**에 따라 결정됩니다.

---

## 3. 옵션: 보류를 완료로 보고 싶을 때

1. **Jira 워크플로우에서 처리 (권장)**  
   - “보류” 상태를 Done 카테고리로 두면, 현재 구조 그대로 완료로 표시됩니다. 코드 변경 불필요.

2. **앱에서 상태명/라벨로 완료로 치환**  
   - 예: `status.name === '보류'` 또는 특정 라벨(예: `on-hold-complete`)이 있으면 `done`과 동일하게 취급하도록,  
     `project-stats-dialog.tsx`·`kpiService.ts` 등에서 `isDone`을 확장하는 방식이 필요합니다.  
   - 이 경우 “보류” 정의(상태명/라벨)를 `jiraConfig`에 두고, 완료 집계·KPI·담당자별 통계에 동일 규칙을 적용해야 일관됩니다.

## 4. 결론

| 항목 | 내용 |
|------|------|
| 보류 → 완료 전용 로직 | **없음** |
| 완료 판단 기준 | **`status.statusCategory.key === 'done'` 만 사용** |
| 현재 동작 | Jira에서 보류가 Done 카테고리면 완료로 표시됨; 그 외 카테고리면 미표시 |
| 보류를 완료로 보고 싶을 때 | Jira에서 Done 카테고리로 두거나, 앱에서 상태명/라벨 기반으로 `done`과 동일 처리하는 확장 필요 |

## 5. 전체 현황에서 보류·취소 포함 이슈 분포 구성

### 5.1 “이슈 분포”가 나오는 두 가지 방식

| 위치 | 기준 | 보류·취소 표시 방식 |
|------|------|----------------------|
| **프로젝트 통계 다이얼로그** → 전체 현황 “이슈 분포” | **statusCategory** 3분할 (완료/진행/대기) | 보류·취소는 **항목으로 안 나옴**. Jira 카테고리에 따라 완료/진행/대기 **중 하나**에만 포함됨. |
| **상태 분포 패널** (`StatisticsPanel`) | **status.name** (상태 이름별) | 보류·취소 **포함**. Jira에 "보류", "취소" 상태가 있으면 각각 **별도 슬라이스**로 집계·표시됨. |

### 5.2 프로젝트 통계 다이얼로그 – 전체 현황

- **진입**: 대시보드 상단 "프로젝트 통계" 버튼 → **프로젝트 현황** 탭.
- **전체 현황**:
  - **전체 이슈 수**: `leafIssues.length` → 보류·취소 이슈 **포함**.
  - **이슈 분포(파이 + 범례)**: `overallSegments` = **[완료, 진행, 대기]** 3개만 사용.
    - `done` = `statusCategory.key === 'done'`
    - `inProg` = `statusCategory.key === 'indeterminate'`
    - `todo` = `statusCategory.key === 'new'` (그 외 모두 여기)
  - 따라서 **“보류” “취소”라는 레이블은 없고**, 해당 이슈들은 위 세 그룹 중 하나에만 들어감.
- **담당자별 현황**:
  - 담당자별 분포도 동일하게 **완료/진행/대기** 3분할(`segs`) + 지연·조기완료·로그 있음/없음 등. 상태명별 구분 없음.
- **그룹 클릭 시 하단 이슈 목록**:
  - 각 행에 **`issue.fields.status.name`** 표시 (546행). 따라서 “완료” 그룹을 열면 그 안에 **보류**, **취소**, **Done** 등 실제 상태명이 같이 노출됨.

**정리**: 프로젝트 통계의 “이슈 분포”는 **카테고리 기준 3분할**이라, 보류·취소를 **별도 항목으로** 보여주지는 않지만, 전체 건수에는 포함되고, 그룹 내 이슈 목록에서는 상태명(보류/취소 포함)이 보임.

### 5.3 상태 분포 패널 (`src/components/statistics-panel.tsx`)

- **집계 방식**: `statusData = leafIssues`를 순회하며 **`i.fields.status.name`** 별로 건수 카운트.
- **표시**: Recharts PieChart로 **상태 이름별** 슬라이스 (예: "완료", "진행 중", "보류", "취소" 등 Jira에 존재하는 모든 상태명).
- **보류·취소**: Jira에 "보류", "취소" 상태가 있으면 **자동으로** 해당 이름으로 집계되어 **이슈 분포에 포함**됨. 별도 설정/필터 없음.

```ts
// statistics-panel.tsx 32–39행
const statusData = useMemo(() => {
    const counts: Record<string, number> = {};
    leafIssues.forEach(i => {
        const status = i.fields.status.name;
        counts[status] = (counts[status] || 0) + 1;
    });
    return Object.entries(counts).map(([name, value]) => ({ name, value }));
}, [leafIssues]);
```

- **참고**: `StatisticsPanel`은 대시보드에서 사용 여부가 라우트/레이아웃에 따라 다를 수 있으나, 코드 상으로는 **전체 리프 이슈**를 상태명별로 나누므로 보류·취소가 있으면 분포에 포함됨.

### 5.4 요약 표

| 구분 | 전체 건수 | 분포 항목(레이블) | 보류·취소 표시 |
|------|-----------|-------------------|----------------|
| 프로젝트 통계 – 전체 현황 이슈 분포 | 포함 | 완료 / 진행 / 대기 (카테고리 3분할) | 분포 레이블로는 안 나옴. 그룹 내 이슈 목록에서는 상태명(보류/취소) 표시. |
| 담당자별 분포 | 포함 | 완료 / 진행 / 대기 + 지연·조기완료 등 | 동일. 상태명별 구분 없음. |
| 상태 분포 패널 (StatisticsPanel) | 포함 | **상태 이름별** (Jira 상태명 그대로) | **포함**. "보류", "취소"가 있으면 각각 별도 슬라이스로 표시됨. |

### 5.5 결론

- **보류·취소 건은 모든 현황에서 전체 이슈 수에는 포함**됨 (리프 이슈 기준).
- **이슈 분포를 “상태 이름별”로 보려면** → **상태 분포 패널**에서만 가능하며, 여기서는 보류·취소가 **자동 포함**됨.
- **프로젝트 통계의 “이슈 분포”**는 완료/진행/대기 **3분할만** 구성하므로, 보류·취소를 **별도 항목**으로 두고 싶다면, 상태명 또는 라벨로 세그먼트를 추가하는 **코드 확장**이 필요함 (예: `overallSegments`에 "보류", "취소" 세그먼트 추가).
