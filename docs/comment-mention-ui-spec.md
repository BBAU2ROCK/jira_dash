# 댓글·멘션 UI 개선: 분석 및 체크리스트

## 1. 요구사항 정리

| 항목 | 내용 |
|------|------|
| 표시 위치 | 활동 이력 중 **댓글** 탭을 클릭했을 때, 해당 탭 **안에서만** 댓글 입력/멘션 UI 표시 |
| 멘션 수정 | 선택한 멘션 대상을 **수정 가능** (제거 또는 다른 사용자로 교체) |
| 멘션 표시 | 멘션을 **입력 박스 내부**에서 칩 형태로 표시 (입력창과 한 영역에 텍스트+멘션 혼합) |

---

## 2. 입력 박스 내 멘션 표시 방안 분석

### 2.1 방안 비교

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. div + span/칩 + input** | 테두리 있는 컨테이너 안에 `[텍스트][@칩][@칩][input]`을 flex로 나열 | 구현 단순, 접근성·키보드 제어 용이, 포커스/커서 예측 가능 | 칩과 input 사이 시각적 구분 필요 |
| **B. contenteditable** | 하나의 contenteditable div에 텍스트+멘션 노드 삽입 | 한 영역에서 자연스러운 인라인 편집 | 커서/선택 처리 복잡, 브라우저별 차이, 붙여넣기/삭제 경계 처리 부담 |
| **C. 숨김 input + 오버레이** | 실제 input은 숨기고, 위에 동일 크기 div로 시각만 표시 | 외형은 단일 입력란처럼 보임 | 클릭→포커스 전달, 커서 위치 동기화 등 구현 복잡도 높음 |

**선택: 방안 A** — 유지보수·동작 예측이 쉽고, 멘션 칩의 수정/삭제를 버튼으로 명확히 할 수 있음.

### 2.2 방안 A 상세

- **데이터**: `segments: (Text | Mention)[]` + `trailingInput: string`
- **렌더**: 테두리 있는 박스(`CommentComposer`) 내부
  - `segments.map`: `text` → `<span>`, `mention` → `<MentionChip />` (표시명, X 버튼, 클릭 시 교체 팝오버)
  - 마지막에 `<input>` (placeholder, value=trailingInput)
- **멘션 추가**: 입력 중 `@` 시 또는 "멘션 추가" 버튼 → 팝오버에서 사용자 선택 시 segment 추가
- **멘션 수정**: 칩 클릭(또는 전용 아이콘) → 같은 사용자 검색 팝오버에서 다른 사용자 선택 시 해당 segment만 교체
- **멘션 삭제**: 칩의 X 버튼 → 해당 segment 제거

## 3. 사이드 이펙트 체크

| 구분 | 항목 | 영향 | 대응 |
|------|------|------|------|
| **탭 전환** | 댓글 탭이 아닐 때는 댓글 입력 UI 비노출 | 전체/기록/업무로그 탭에서는 작성 중이던 내용이 안 보임 | 댓글 탭에 들어올 때만 composer 표시. 필요 시 `commentSegments`/`commentInputValue`를 전역(또는 drawer 스코프)으로 유지해 탭 이동 후 복귀 시 복원 가능 (현재는 탭 이동 시에도 state 유지) |
| **포커스** | 댓글 탭으로 전환 시 input 포커스 자동 여부 | 자동 포커스 시 스크롤/키보드 올라옴 | 기본값: 자동 포커스 안 함. 필요 시 옵션으로만 제공 |
| **키보드** | Enter로 전송 시 탭 내 다른 버튼/링크와 충돌 | 폼 내부이므로 Enter → submit 동작만 유지 | `onKeyDown`에서 Enter 시 `preventDefault` + 등록 처리 (기존과 동일) |
| **접근성** | 칩 제거/수정이 키보드만으로 가능해야 함 | 스크린 리더, 키보드 사용자 | 칩에 `button` + `aria-label`, 포커스 가능하도록 구현 |
| **반응형** | 좁은 너비에서 칩+input 줄바꿈 | flex-wrap으로 자연스럽게 줄바꿈 | `flex flex-wrap` + input `min-width` 유지 |
| **에러 상태** | 댓글 등록 실패 시 입력 내용 유지 | 이미 구현됨 | mutation 실패 시 segments/input 초기화하지 않음 |
| **다중 멘션** | 동일 사용자 여러 번 멘션 | ADF/API는 동일 사용자 여러 번 허용 | 제한 두지 않음 (현재와 동일) |

## 4. 기능 개발 체크리스트

### 4.1 표시 위치

- [x] 활동 탭 중 **댓글** 탭을 클릭했을 때만 댓글 입력 영역 표시
- [x] 전체/기록/업무로그 탭에서는 댓글 입력 영역 비표시
- [x] 댓글 탭 내부 상단에 "댓글 입력 UI" → 하단에 "댓글 목록" 배치

### 4.2 입력 박스 내 멘션 표시

- [x] 하나의 테두리 있는 입력 박스(컨테이너) 안에 **텍스트 구간 + 멘션 칩 + 입력 필드**가 함께 표시
- [x] 멘션 칩은 `@displayName` 형태, 입력 필드와 같은 줄(또는 wrap)에 인라인 배치
- [x] `@` 입력 시 사용자 검색 팝오버 표시, 선택 시 칩으로 추가
- [x] "멘션 추가" 버튼으로도 동일하게 팝오버 → 선택 시 칩 추가

### 4.3 멘션 수정

- [x] 멘션 칩 클릭 시 **사용자 검색 팝오버** 오픈 (해당 칩만 교체 모드)
- [x] 다른 사용자 선택 시 해당 칩만 새 사용자로 **교체**
- [x] 칩에 **X 버튼** 제공 시 해당 멘션 **삭제**

### 4.4 기존 동작 유지

- [x] 댓글 등록 시 ADF 생성 및 `addComment` API 호출 동일
- [x] 등록 성공 시 `issueDetails` / `issues` 쿼리 무효화 및 입력 초기화
- [x] 등록 실패 시 에러 메시지 표시, 입력 내용 유지

### 4.5 접근성·UX

- [x] 멘션 칩 제거/수정 버튼에 `aria-label` 적용 (`멘션 {이름} 수정`, `멘션 {이름} 제거`)
- [ ] 키보드만으로 칩 포커스 및 제거/수정 가능 (선택 사항, 추후 개선)

---

## 5. 구현 순서 제안

1. 댓글 입력 블록을 **댓글 탭** `TabsContent` 안으로 이동 (다른 탭에서는 미표시).
2. 기존 세그먼트+input UI를 **단일 입력 박스**로 리팩터: 컨테이너 내부에 `[세그먼트들] + input`.
3. 멘션 세그먼트를 **칩**으로 렌더, 칩에 **X(삭제)** 버튼 추가.
4. 칩 클릭(또는 수정 아이콘) 시 **사용자 검색 팝오버**로 해당 멘션만 **교체**하는 로직 추가.
5. 접근성 및 좁은 화면에서 줄바꿈 확인.
