## 4. 새로운 결정 사항 (Phase 0 결과 기반)

### 4.1 분석 대상 프로젝트 변경 권장

현재 `JIRA_CONFIG.DASHBOARD.PROJECT_KEY = 'IGMU'`는 본 신규 기능에 부적합 (백로그 4건). **3가지 옵션**:

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. PROJECT_KEY를 IPCON으로 변경** | jiraConfig 한 줄 수정 | 즉시 활성 백로그 분석 | 기존 사용자 화면 변동 |
| **B. 다중 프로젝트 지원 (Tier 2 확장)** | 사이드바에 프로젝트 선택 추가 | 유연성 | 작업 +0.5일 |
| **C. PROJECT_KEY 유지 + 신규 기능 탭에서만 프로젝트 선택 가능** | 기존 화면은 IGMU, 진행 추이는 선택 가능 | 영향 최소 + 유연 | UI 약간 복잡 |

→ **권장: C** (영향 최소 + 검증 가능)

### 4.2 hybrid 공수 추정 단순화

원래 4가지 데이터 소스 (worklog/SP/난이도/cycle time) hybrid였으나, 측정 결과 SP/난이도가 0%:

**수정안**: 우선순위 단순화
1. **Worklog** (있으면 — IGMU에서만)
2. **Cycle time fallback** (항상)

→ effortEstimation.ts 코드는 그대로 두되, 실제 분기는 2가지만 의미 있음. 사용자 환경에 따라 자동 결정.

### 4.3 Tier 일정 영향 (옵션 C 채택 시)

- 원안 4일 → **+0.5일 (다중 프로젝트 선택 UI)** = 4.5일
- 또는 옵션 A 채택 시 4일 그대로

### 4.4 신뢰도 등급의 첫 검증

본 측정으로 **신뢰도 등급 알고리즘이 정직하게 작동함을 사전 검증**:
- IGMU → low (단일 ETA 숨김)
- IPCON → unreliable (강한 경고)

→ 본 기능이 거짓 약속을 만들지 않음을 데이터로 입증.
